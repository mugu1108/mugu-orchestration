# agents/**/*.md テンプレート

## フロントマター

orchestrator.md のみフロントマターあり:

```yaml
---
name: agent-name
description: エージェントの説明
model: sonnet
tools: Read, Glob, Task
---
```

`agents/development/` や `agents/business/` 配下のエージェントはフロントマターなし。

## 構造

```markdown
# エージェント名 Agent - 役割説明

## 概要

エージェントの目的と設計思想を2〜3文で。
必要に応じて引用ブロックで設計原則を示す。

> 設計思想の引用（任意）

## 使用タイミング

1. **ケース1**: 説明
2. **ケース2**: 説明

## ワークフロー / レビュー項目 / 計画作成プロセス

（エージェント固有のメインセクション）

## レポートのフォーマット / 出力フォーマット

（Markdownコードブロックでテンプレート提示）

## mugu-orchestration固有のコンテキスト

- プロジェクト固有の基準・設定

## 使用例

### 例1: タイトル
具体的な入出力

## ベストプラクティス

1. 番号付きリスト

## チェックリスト

- [ ] チェック項目

## 他のエージェントとの連携

- **エージェント名**: 連携内容

## モデル選択

- **推奨モデル**: 理由

## コマンド

呼び出し方

## 注意事項

- 制約や注意点
```

## 必須セクション

1. H1タイトル（`エージェント名 Agent - 役割` 形式）
2. 概要
3. ワークフロー or メインセクション
4. 使用例（最低1つ）

## agent / skill / command の役割分担

agent.md がドメイン知識の Single Source of Truth:

| 書く場所 | 内容 |
|---------|------|
| **agent.md** | レビュー項目、出力フォーマット、プロジェクト固有基準（定義元） |
| **SKILL.md** | ワークフロー（エージェント呼び出し手順）、`agent.md の指示に従って` で参照 |
| **commands/** | ユーザー向けリファレンス（使い方・使用例のみ） |

- レビュー項目やフォーマットを3ファイルにコピーしない。agent.md に1箇所で定義し、他は参照する

## ベストプラクティス / チェックリストの書き分け

- **Claudeへの行動指示** → ワークフローやレビュー項目に含める
- **人間向けの運用手順**（PR前チェック等） → `commands/*.md` に書くか、削除する
- 「建設的なフィードバック」「学習機会」等の一般論は不要（Claudeは自明に理解する）

## ガイドライン

- `agents/development/` や `agents/business/` 配下はフロントマターなし
- orchestrator のみフロントマターあり
- Slack応答例はビジネスエージェントで効果的
- 機能一覧テーブル（`| 機能 | ステータス |`）は実装状況の管理に有用
- フロー図はテキストの矢印（`↓`）で簡潔に
- `## モデル選択` はスキルの性質に応じた具体的な判断基準を書く
